## 1. 概要

### 1-1. アプリ名・命名由来

**アプリ名**: papazon-dash

**命名由来**: papa（パパ）+ Amazon Dash（ワンタッチで商品を注文するデバイス）を掛け合わせた造語。「妻がワンタッチでパパに買い物を頼める」というコアコンセプトを体現する名称。

### 1-2. PoCコンセプト

papazon-dash は、夫婦・カップルの 2 人限定を前提とした買い物依頼・リマインドアプリである。

**解決課題（Lean Canvas Problem）**: 「何度言っても買ってこない夫」に対し、妻がリマインドを毎日口頭でしなくて済む状態を実現する。

**コアバリュー（Unique Value Proposition）**: 依頼者（マスター=妻想定）がワンタップでお使いを登録・通知し、実行者（スレーブ=夫想定）の行動率を向上させることで、二度手間・無言の怒りを解消する。

**差別化ポイント（Wave0調査結果より）**:
- 既存アプリ（Bring!・AnyList・Cozi等）は家族全員向けで権限設計が複雑。papazon-dash は 1 対 1 固定ロールに特化し、導線とデータモデルを極限までシンプルにする。
- 「お使い（Errand）」という単一イベントに絞り込み、「夫が無視できない通知 UX」（インタラクティブ通知・コンテキスト型リマインド）を標準装備する。

## 3. アクター

| ロール | 別称 | 想定ユーザー | 権限概要 |
|--------|------|--------------|----------|
| **マスター** | 依頼者 | 妻（女性パートナー）想定 | アイテム作成・編集・削除・リマインド再送 |
| **スレーブ** | 実行者 | 夫（男性パートナー）想定 | アイテム閲覧・完了報告 |

**ロール設計方針**:
- ロールはペアリング時に「招待を送った側＝マスター」「招待を受けた側＝スレーブ」として自動決定し、固定する。
- PoC ではロール変更機能を提供しない（バックログ参照）。
- 「想定ユーザー」は統計的傾向に基づく表現であり、実際のジェンダー制限はない。

## 4. 業務機能

### 業務機能 1: リスト管理（アイテム CRUD）

**概要**: マスターが買い物依頼アイテムを作成・編集・削除する中核機能。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-1-1 | アイテム作成 | マスター |
| UC-1-2 | アイテム編集 | マスター |
| UC-1-3 | アイテム削除 | マスター |
| UC-1-4 | アイテム一覧表示 | マスター / スレーブ |

**UC-1-1 アイテム作成 基本フロー**:
1. マスターがホーム画面の「＋」ボタンをタップ
2. アイテム名を入力（テキストのみ、数量は任意テキスト入力）
3. 「依頼する」をタップ
4. `errands` コレクションに新規ドキュメント作成（status: `pending`）
5. 業務機能 6（FCM 通知）が連携してスレーブに通知送信

#### 要件

| ID | 要件 |
|----|------|
| F-LIST-01 | アイテムの新規作成（マスターのみ） |
| F-LIST-02 | アイテムタイトルの編集（マスターのみ、`completed` 以外のアイテムに限る） |
| F-LIST-03 | アイテムの削除（マスターのみ、`completed` / `pending` 両方対象） |
| F-LIST-04 | 未完了アイテムの一覧表示（マスター・スレーブ両者） |
| F-LIST-05 | 完了済みアイテムの履歴タブへの自動移動（status `completed` 検知時） |

#### データフロー

```
マスター入力
  → Firestore errands/{errandId} 新規書き込み (status: pending)
  → snapshotListener (業務機能2) が両者画面を即時更新
  → FCM トリガー (業務機能6) がスレーブに push 配信
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| ホーム（マスター） | 未完了アイテム一覧 + 「＋」FAB |
| ホーム（スレーブ） | 未完了アイテム一覧（読み取り専用） |
| 入力ダイアログ | アイテム名テキストフィールド + 「依頼する」「キャンセル」ボタン |

#### Firestore操作

```
コレクション: errands
書き込み (作成):
  errandId:   UUID (自動生成)
  pairId:     ペアID
  title:      アイテム名
  status:     "pending"
  createdBy:  マスター UID
  assignedTo: スレーブ UID
  partners:   [masterUid, slaveUid]  ← セキュリティルール用
  createdAt:  Timestamp
  updatedAt:  Timestamp

更新 (編集): title / updatedAt
削除: ドキュメント削除
```

### 業務機能 2: リストのリアルタイム共有（データ層）

> **注意 — 機能 2 vs 機能 6 の差異**: 機能 2 は「Firestore のデータ変更を双方の画面に即時反映するデータ同期層」。機能 6 は「依頼イベントをスレーブ端末に push 通知で届ける通知層」。詳細は §6（機能2/6差異明示）参照。

**概要**: Firestore の `addSnapshotListener` を用い、マスター・スレーブ双方の画面がデータ変更を無操作で即時反映する。アプリがフォアグラウンドにある状態での双方向同期が主目的。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-2-1 | リスト追加の即時反映（スレーブ画面） | システム |
| UC-2-2 | 完了状態の即時反映（マスター画面） | システム |
| UC-2-3 | オフライン時ローカルキャッシュ参照 | システム |
| UC-2-4 | ネットワーク復帰後の自動同期 | システム |

#### 要件

| ID | 要件 |
|----|------|
| F-SYNC-01 | `addSnapshotListener` によるリアルタイムリスト更新（マスター・スレーブ両者） |
| F-SYNC-02 | オフライン時はローカルキャッシュを参照し、復帰時に自動同期（Firebase Android SDK デフォルト Persistence を利用） |
| F-SYNC-03 | オフライン中は UI バナーで「オフライン中」表示 |
| F-SYNC-04 | 同期失敗時（認証切れ等）はエラートーストを表示 |

#### データフロー

```
Firestore errands コレクション変更
  → snapshotListener コールバック発火
  → Kotlin Flow / StateFlow 経由で ViewModel に伝播
  → Jetpack Compose の collectAsState() でリスト画面を再コンポーズ
  → 両ユーザーの画面が無操作で自動更新
```

#### 画面

| 画面 | 動的更新内容 |
|------|------------|
| ホーム（双方） | アイテム追加・削除・status変更が即時反映 |
| オフラインバナー | ネットワーク切断時に「オフライン中 - 復帰時に同期します」表示 |

#### Firestore操作

```kotlin
// callbackFlow を用いたリアルタイム監視（Wave0 §2-3 実装例準拠）
fun getErrandsFlow(pairId: String): Flow<List<Errand>> = callbackFlow {
    val query = Firebase.firestore.collection("errands")
        .whereEqualTo("pairId", pairId)
    val listener = query.addSnapshotListener { snapshot, error ->
        if (error != null) { close(error); return@addSnapshotListener }
        snapshot?.let { trySend(it.toObjects(Errand::class.java)) }
    }
    awaitClose { listener.remove() }
}
```

### 業務機能 3: スレーブ側リマインド（繰り返し・スヌーズ）

**概要**: 依頼後に未完了が続く場合、自動または手動でスレーブに再通知を送る機能。「夫が忘れる」問題への直接的な解決手段。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-3-1 | 自動リマインド送信（N時間後） | システム |
| UC-3-2 | マスター手動リマインド送信 | マスター |
| UC-3-3 | スレーブによるスヌーズ（後で通知） | スレーブ |
| UC-3-4 | リマインド上限到達の通知 | システム |

**UC-3-1 自動リマインド 基本フロー**:
1. 依頼作成から N 時間（初期値 3h）が経過
2. `errands/{id}.status` が `pending` または `acknowledged` のまま
3. システムがスレーブへ FCM リマインド通知を送信
4. `remindCount` をインクリメント、`lastRemindAt` を更新
5. `remindCount` が上限（初期値 5）に達したらマスターに「リマインド上限到達」通知

#### 要件

| ID | 要件 |
|----|------|
| F-REMIND-01 | 依頼後 N 時間（初期値 3h、マスターが 1〜24h で設定可）未完了の場合に自動再通知 |
| F-REMIND-02 | 自動リマインドの上限回数（初期値 5 回） |
| F-REMIND-03 | マスターによる手動リマインド送信（アイテム長押しメニューから） |
| F-REMIND-04 | リマインド送信履歴をアイテム詳細に表示（「3 回リマインド済み」等） |
| F-REMIND-05 | スレーブが通知の「後で」をタップ → スヌーズ時間（15分固定）後に再通知 |
| F-REMIND-06 | サイレント時間帯（初期値 22:00〜07:00）は自動リマインドを停止 |

#### データフロー

```
タイマー（Firebase Functions or ローカルWorkManager）
  → errands クエリ: pairId一致 & status IN [pending, acknowledged]
               & updatedAt < now - reminderInterval
  → FCM Data Message 送信（業務機能6と共通FCMチャネルを利用）
  → errands/{id} の remindCount++, lastRemindAt 更新
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| 通知（スレーブ） | 「[アイテム名]を買いましたか？（リマインド N回目）」+ 「了解」「後で（15分）」「無理」アクション |
| アイテム詳細 | リマインド回数・最終リマインド日時 |

#### Firestore操作

```
更新フィールド（リマインド送信時）:
  errands/{errandId}:
    remindCount:  +1
    lastRemindAt: Timestamp
```

### 業務機能 4: チェックリスト（完了状態トグル）

**概要**: スレーブが実際にお使いを完了した際、アイテムを完了状態にマークする操作。最もシンプルかつ頻度が高いアクション。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-4-1 | アイテム完了マーク | スレーブ |
| UC-4-2 | 完了取り消し（誤操作対応） | スレーブ |
| UC-4-3 | 通知インタラクションからの完了マーク | スレーブ |

**UC-4-1 基本フロー**:
1. スレーブがリスト画面でアイテムを確認
2. アイテムをタップ → 詳細画面 or インラインの「買ってきた！」ボタンをタップ
3. `errands/{id}.status` を `completed` に更新
4. 業務機能 5（完了通知）がトリガーされマスターに通知送信
5. アイテムが「完了済み」セクションへ移動（snapshotListenerで即時反映）

#### 要件

| ID | 要件 |
|----|------|
| F-CHECK-01 | アイテムタップ or ボタンタップによる完了マーク（スレーブのみ） |
| F-CHECK-02 | 完了後 5 分以内は「取り消し」操作を許可（誤タップ対応） |
| F-CHECK-03 | 通知アクション「今買った」からもアプリ起動なしで完了マーク可能 |
| F-CHECK-04 | 完了アイテムは即時にリストから履歴セクションへ移動 |

#### データフロー

```
スレーブ操作（ボタンタップ or 通知アクション）
  → errands/{id}.status = "completed", completedAt = Timestamp
  → snapshotListener (機能2) がマスター・スレーブ双方の画面を更新
  → 完了通知 (機能5) がマスターに FCM 送信
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| ホーム（スレーブ） | 各アイテム行に「買ってきた！」ボタン（チェックアイコン） |
| 完了直後 | 「取り消す」スナックバー（5秒表示） |

#### Firestore操作

```
更新フィールド（完了マーク時）:
  errands/{errandId}:
    status:      "completed"
    completedAt: Timestamp
    updatedAt:   Timestamp

更新フィールド（完了取り消し時、5分以内）:
  errands/{errandId}:
    status:      "acknowledged"  // 直前のstatusに戻す
    completedAt: null
    updatedAt:   Timestamp
```

### 業務機能 5: 完了通知 / 完了履歴（スレーブ完了→マスター通知・履歴蓄積）

**概要**: スレーブが完了マークをつけた際に、マスターへ完了を通知し、完了済みアイテムを履歴として蓄積する。マスターの「ちゃんと買ってきたか確認できない」問題を解消。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-5-1 | 完了通知受信（マスター） | システム |
| UC-5-2 | 完了履歴一覧閲覧 | マスター / スレーブ |
| UC-5-3 | 履歴アイテム詳細確認 | マスター / スレーブ |

**UC-5-1 完了通知受信 基本フロー**:
1. スレーブが業務機能 4 でアイテムを完了マーク
2. システムがマスターに FCM 完了通知を送信（アイテム名・完了日時含む）
3. マスターが通知を受信、アプリ画面で「完了済み」セクションにアイテムが表示
4. 履歴タブにアイテムが追加（依頼日時・完了日時・リマインド回数付き）

#### 要件

| ID | 要件 |
|----|------|
| F-HIST-01 | スレーブ完了時にマスターへ FCM 完了通知送信 |
| F-HIST-02 | 完了済みアイテムを最大 30 件表示（新しい順）|
| F-HIST-03 | 各アイテムに「依頼日時」「完了日時」「リマインド回数」を表示 |
| F-HIST-04 | 完了から 90 日後に自動削除（Firestore TTL or スケジュール処理） |
| F-HIST-05 | 履歴は読み取り専用（マスター・スレーブとも削除・編集不可） |

#### データフロー

```
errands/{id}.status = "completed" (機能4書き込み)
  → snapshotListener (機能2) がリスト画面の完了セクションを更新
  → FCM 完了通知 (機能6のチャネル共有) をマスターに送信
  → 履歴タブクエリ: pairId一致 & status = "completed" を createdAt 降順で最大30件取得
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| 通知（マスター） | 「[スレーブ名]が[アイテム名]を買ってきました！」 |
| 履歴タブ | 完了アイテム一覧（依頼日時・完了日時・リマインド回数） |
| アイテム詳細 | 全ステータス変遷タイムライン |

#### Firestore操作

```
読み取り（履歴取得）:
  errands コレクション
    .whereEqualTo("pairId", pairId)
    .whereEqualTo("status", "completed")
    .orderBy("completedAt", DESCENDING)
    .limit(30)
```

### 業務機能 6: 指令送信時リアルタイム送信（FCM push・通知層）

> **注意 — 機能 2 vs 機能 6 の差異**: 機能 6 は「依頼イベントをスレーブ端末に push 通知で届ける通知層」。機能 2 は「Firestore のデータ変更を双方の画面に即時反映するデータ同期層」。詳細は §6（機能2/6差異明示）参照。

**概要**: マスターがアイテムを作成した瞬間に、FCM high-priority プッシュ通知でスレーブ端末に「今この依頼が届いた」というイベントを配信する。アプリがバックグラウンドまたは終了状態でも届く通知層。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-6-1 | 新規依頼通知送信（スレーブへ） | システム（マスター作成をトリガー） |
| UC-6-2 | 完了通知送信（マスターへ） | システム（スレーブ完了をトリガー） |
| UC-6-3 | 通知アクション応答（アプリ起動不要） | スレーブ |
| UC-6-4 | Doze Mode 突破（high-priority 設定） | システム |

**UC-6-1 基本フロー**:
1. マスターがアイテムを Firestore に書き込み（業務機能 1）
2. Firebase Functions（または Android WorkManager）が書き込みを検知
3. FCM Data Message（priority: high）をスレーブの FCM トークン宛に送信
4. スレーブ端末の通知領域に「[アイテム名]を買ってきて！」表示
5. 通知にインタラクティブアクション（「了解」「後で」「無理」）を配置

#### 要件

| ID | 要件 |
|----|------|
| F-NOTIF-01 | FCM `high` priority による新規依頼プッシュ通知（Doze Mode 突破） |
| F-NOTIF-02 | 通知ペイロードにアイテム名・依頼者名・errandId を含める |
| F-NOTIF-03 | インタラクティブ通知アクション（「了解」「後で 15分」「無理」）の実装 |
| F-NOTIF-04 | アクションタップ時のFirestore ステータス更新（アプリ起動不要） |
| F-NOTIF-05 | 完了報告時のマスターへの完了通知送信 |
| F-NOTIF-06 | FCM トークン管理（アプリ起動時にトークンを users/{uid}.fcmToken へ保存） |

#### データフロー

```
マスター: errands/{id} 新規書き込み
  → Firebase Functions onDocumentCreated トリガー
    （または WorkManager + Firestore trigger）
  → FCM Admin SDK: sendToDevice(slaveFcmToken, payload)
    payload.priority = "high"
    payload.data = { errandId, title, masterName }
  → スレーブ端末: 通知バー表示 + アクションボタン
  → アクションタップ: BroadcastReceiver が Firestore status 更新
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| 通知（スレーブ） | タイトル: 「[マスター名]からの依頼」本文: 「[アイテム名]を買ってきて！」 アクション: 「了解」「後で」「無理」 |
| 通知（マスター・完了時） | 「[スレーブ名]が[アイテム名]を買ってきました！」 |

#### Firestore操作

```
読み取り（FCM トークン取得）:
  users/{slaveUid}.fcmToken

更新（アクション応答時）:
  errands/{errandId}:
    status: "acknowledged" (了解) / "snoozed" (後で) / "rejected" (無理)
    updatedAt: Timestamp
```

## 5. 基盤機能

### 基盤機能 A: ユーザー認証（Firebase Auth）

**概要**: Google Sign-In を用いたサインイン・アカウント管理。若年層の心理的障壁を下げるためワンタップ認証を採用。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-A-1 | Google アカウントでサインイン / 新規登録 | マスター / スレーブ |
| UC-A-2 | サインアウト | マスター / スレーブ |
| UC-A-3 | アカウント削除（退会） | マスター / スレーブ |

**UC-A-1 基本フロー**:
1. 初回起動 → サインイン画面表示
2. 「Google でサインイン」ボタンをタップ
3. Google アカウントピッカー表示 → 選択
4. Firebase Auth が UID 発行、`users/{uid}` ドキュメント作成
5. 未ペアリングの場合 → ペアリング画面へ遷移

#### 要件

| ID | 要件 |
|----|------|
| F-AUTH-01 | Google Sign-In によるサインイン / アカウント新規作成 |
| F-AUTH-02 | Firebase Auth UID を主キーとしたユーザー管理 |
| F-AUTH-03 | サインアウト機能（ペア接続は維持） |
| F-AUTH-04 | アカウント削除（退会）機能 ※ペア解除・データ削除を伴う |
| F-AUTH-05 | 未認証状態でのアプリ操作を全画面でブロック（AuthState 監視） |

#### データフロー

```
Google Sign-In 成功
  → Firebase Auth: UID 発行・ID トークン発行
  → users/{uid} 作成 or 更新:
      { uid, displayName, email, pairId(null), role(null), fcmToken }
  → pairId 有 → ホーム画面
  → pairId 無 → ペアリング画面（基盤機能B）
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| サインイン | 「Google でサインイン」ボタン + アプリロゴ・キャッチコピー |
| 設定 > アカウント | メールアドレス・UID表示 / サインアウト / アカウント削除ボタン |

#### Firestore操作

```
コレクション: users
ドキュメント: users/{uid}
フィールド:
  uid:         String
  displayName: String
  email:       String
  pairId:      String | null
  role:        "master" | "slave" | null
  fcmToken:    String | null
  createdAt:   Timestamp
```

### 基盤機能 B: 夫婦ペアリング（招待コード or QR）

**概要**: マスターが発行した 6 桁ワンタイム招待コードを介して、1 対 1 のペアを形成する。Firestore Transaction で排他制御し、3 人目の接続を防ぐ。

#### ユースケース

| UC | タイトル | 主アクター |
|----|---------|----------|
| UC-B-1 | 招待コード発行 | マスター（招待送信側） |
| UC-B-2 | 招待コード入力・ペア接続 | スレーブ（招待受信側） |
| UC-B-3 | ペア解除 | マスター / スレーブ |

**UC-B-1 〜 B-2 基本フロー**:
1. マスターが「パートナーを招待」を選択 → 6 桁コード生成（有効期限 24 時間）
2. LINE / SMS 等でコードをスレーブに共有
3. スレーブがアプリ内でコード入力 → 「ペアに参加」
4. Firestore Transaction: `invitations/{code}` を取得 → 期限確認 → `pairs/{pairId}` 作成 → 両者の `users/{uid}.pairId / role` を更新
5. 双方の画面にペアリング完了メッセージ表示

#### 要件

| ID | 要件 |
|----|------|
| F-PAIR-01 | 6 桁ワンタイム招待コード生成（有効期限 24 時間） |
| F-PAIR-02 | 招待コード入力によるペア接続（Firestore Transaction で排他制御、1 コードにつき最大 2 人） |
| F-PAIR-03 | ペアリング後のロール固定（マスター / スレーブ変更不可） |
| F-PAIR-04 | ペア解除機能（双方の pairId / role をリセット、errands データは 30 日保持後削除） |
| F-PAIR-05 | 既にペアリング済みのユーザーは新規ペアリング不可 |
| F-PAIR-06 | コード期限切れ / 3 回入力ミスでエラー表示 → マスターに再発行を促す |

#### データフロー

```
マスター: コード生成
  → invitations/{code} 作成: { code, masterUid, pairId(新UUID), expiresAt, used: false }

スレーブ: コード入力
  → Firestore Transaction:
      1. invitations/{code} 取得 → expiresAt 確認, used 確認
      2. pairs/{pairId} 作成: { pairId, partners: [masterUid, slaveUid] }
      3. users/{masterUid}.pairId = pairId, role = "master"
      4. users/{slaveUid}.pairId = pairId, role = "slave"
      5. invitations/{code}.used = true
```

#### 画面

| 画面 | 表示内容 |
|------|---------|
| 招待コード発行 | 6桁コード大文字表示 + コピーボタン + 有効期限カウントダウン |
| コード入力 | 6桁入力フィールド + 「参加する」ボタン |
| ペアリング完了 | 「[相手の表示名]とペアになりました！」+ ホームへボタン |

#### Firestore操作

```
コレクション: invitations
  invitations/{code}: { code, masterUid, pairId, expiresAt, used }

コレクション: pairs
  pairs/{pairId}: { pairId, partners: [masterUid, slaveUid], createdAt }

更新: users/{uid}.pairId, role
```

**概要**: ユーザーが通知挙動やリマインド間隔を自分の生活リズムに合わせて調整できる設定画面。PoC での設定項目は最小限。

## 6. 機能 2 / 機能 6 差異明示

### 概念的差異

| 観点 | 業務機能 2（データ層） | 業務機能 6（通知層） |
|------|----------------------|---------------------|
| **役割** | Firestoreデータの双方向リアルタイム同期 | 依頼イベントのスレーブ端末への push 配信 |
| **技術** | Firestore `addSnapshotListener` + Kotlin Flow | Firebase Cloud Messaging（FCM）Data Message |
| **起動条件** | アプリがフォアグラウンドにある状態で機能 | アプリがバックグラウンド・終了状態でも機能（Doze Mode 突破） |
| **到達保証** | Firestoreへの接続が維持されている間のみ | FCM high-priority で OS レベルで強制配信 |
| **双方向性** | マスター↔スレーブ 双方向データ同期 | 依頼: マスター→スレーブ 一方向 / 完了: スレーブ→マスター 一方向 |
| **データの型** | Firestoreドキュメントの変更差分（データ） | イベント通知ペイロード（タイトル・メッセージ・errandId） |

### シナリオ別動作比較

| シナリオ | 機能 2 の動作 | 機能 6 の動作 |
|---------|--------------|--------------|
| マスターがアイテム追加（スレーブはアプリ開き中） | スレーブのリスト画面が即時更新 | スレーブに push 通知が届く |
| マスターがアイテム追加（スレーブはアプリ終了中） | **動作しない**（リスナー不活性） | **push 通知が届く**（FCM high-priority） |
| スレーブがアイテム完了（マスターはアプリ開き中） | マスターのリスト画面が即時更新 | マスターに完了 push 通知が届く |
| オフライン状態のアイテム追加 | ローカルキャッシュに書き込み → 復帰後同期 | オフライン中は通知遅延（FCM 制約） |

### 実装上の依存関係

```
機能 1（アイテム作成）
  ├── 機能 2（snapshotListener）: Firestore 書き込みを検知 → 画面更新
  └── 機能 6（FCM）: アイテム作成イベントをトリガー → push 通知送信

⇒ 両者は独立した層として並存し、互いを代替できない。
   フォアグラウンドでの UX = 機能 2 が担い、
   バックグラウンド到達性 = 機能 6 が担う。
```

## 7. 非機能要件

### 7-1. オフライン耐性

| 項目 | 要件 |
|------|------|
| ローカルキャッシュ | Firebase Android SDK のデフォルト Persistence を有効化（明示的コード不要） |
| 書き込みキュー | オフライン中の操作はキューに積み、復帰時に自動フラッシュ |
| 通知受信 | オフライン中は受信遅延が発生しうる旨を UI で明示 |

### 7-2. 通知タイミング

| 項目 | 要件 |
|------|------|
| 即時性 | 新規依頼作成から通知受信まで原則 5 秒以内（FCM high-priority + Doze Mode 突破） |
| リマインド間隔 | 初期値 3 時間。マスターが 1〜24 時間の範囲で設定可能 |
| サイレント時間帯 | 22:00〜07:00 はリマインド自動送信を停止（設定変更可） |

### 7-3. PoC 規模制限

| 項目 | 要件 |
|------|------|
| 同時ペア数 | PoC 期間中は 100 ペア以内を想定（Firestore Free Tier 範囲内） |
| アイテム上限 | 1 ペアあたり未完了アイテム最大 50 件 |
| 履歴保持期間 | 完了から 90 日間（以降は自動削除） |

### 7-4. UI/UX（Wave0調査 §4 準拠）

| 項目 | 要件 |
|------|------|
| タップターゲット | 主要ボタンは最小 48×48 dp |
| フォントサイズ | 本文 16sp 以上、コントラスト比 4.5:1 以上（WCAG 2.1） |
| 主要アクション配置 | 「依頼作成」「完了」ボタンは画面下部中央（Thumb Zone） |

### 7-5. 制約事項

| # | 制約 | 根拠 |
|---|------|------|
| C-01 | ペア人数は厳密に 2 人まで（拡張禁止） | PO requirement:「2 人限定」 |
| C-02 | ロールはペアリング時に固定し、PoC 中に変更不可 | PO requirement:「役割固定」 |
| C-03 | Android アプリのみ対応（iOS / Web は PoC 対象外） | PoC スコープ |
| C-04 | 収益化機能（課金・広告）はコード上も含めない | PO requirement:「収益スコープ外」 |
| C-05 | applicationId は `com.smartse.papazon_dash` または `jp.maya.papazondash`（Phase 2 設計書で確定） | task YAML app_naming |
| C-06 | アプリ名表記は全文 **papazon-dash** に統一（「お使いリマインドアプリ」等の別称使用禁止） | Confirmed by PO 2026-06-13 |
